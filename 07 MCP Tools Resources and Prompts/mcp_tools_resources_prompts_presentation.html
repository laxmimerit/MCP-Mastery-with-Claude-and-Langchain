MCP Framework

Tools, Resources & Prompts

Understanding the three core capabilities of the Model Context Protocol: Tools for execution, Resources for knowledge, and Prompts for guidance

Introduction to MCP Capabilities

The Three Pillars of MCP

  • Tools: Model-controlled actions - "Do something now" pattern
  • Resources: Application-controlled context - "Here's information" pattern
  • Prompts: User-controlled interactions - "Here's how to think" pattern

Each component serves a distinct purpose in creating powerful, modular AI workflows through coordinated interaction patterns.

Resources: The Knowledge Layer

What Are Resources?

Resources are read-only interfaces that expose data as structured, contextual information. They act as intelligent knowledge bases, allowing models to access and reason about information without causing side effects.

Note: In programming, a "side effect" refers to any observable change in system state beyond simply returning a value - like writing files, updating databases, or making API calls.

Resource Types

  • Text Resources: UTF-8 encoded text data
  • Binary Resources: Base64-encoded binary data
  • Static Resources: Fixed URI and content
  • Dynamic Resources: URI templates with parameters

Common Use Cases

  • File contents and documentation
  • API cached responses
  • Database snapshots
  • Configuration files
  • System logs and documentation

Static Resource Implementation

Static Resource Example
@mcp.resource("file://about")
def get_about_info() -> str:
    """Reads and returns contents of a static text file."""
    with open("about.txt", "r", encoding="utf-8") as f:
        return f.read()

Dynamic Resource Templates

Dynamic Resource with URI Template
@mcp.resource("file://{path}/data.json")
def read_file(path: str) -> str:
    """Read the contents of a file at the given path."""
    with open(f"{path}/data.json", 'r') as f:
        return f.read()

Resource templates follow RFC 6570 standards and are particularly powerful for scalable implementations. Instead of registering thousands of individual files, a server can expose a single template that covers entire families of resources.

Resource Metadata Customization

Resource with Full Metadata
@mcp.resource(
    uri="data://app-status",
    name="ApplicationStatus",
    description="Provides the current status of the application.",
    mime_type="application/json",
    tags={"monitoring", "status"}
)
def get_application_status() -> dict:
    """
    Internal function description docstring
    (this is ignored if description is provided above)
    """
    return {
        "status": "ok",
        "uptime": 12345,
        "version": "1.2.0"
    }

Duplicate Resource Management

FastMCP Server Configuration
mcp = FastMCP(
    name="SampleServer",
    on_duplicate_resources="error"  # Raise error on duplicates
)

Duplicate Behavior Options

  • "warn" (default): Logs a warning, new resource replaces old one
  • "error": Raises ValueError, prevents duplicate registration
  • "replace": Silently replaces existing resource with new one
  • "ignore": Keeps original resource, ignores new registration

Prompts: The Guidance Layer

Understanding Prompts in MCP

Prompts are predefined message templates that encapsulate reusable instructions for LLMs. They serve as instructional blueprints, domain-specific reasoning setups, and multi-turn conversational flows that clients can explicitly invoke.

Key Insight: Prompts are selected by the user and set the context before the model begins generating

Why Use Prompts?

  • Centralized, server-defined strategies
  • Eliminates manual prompt writing
  • Discoverable via /prompts/list
  • Server-side updates possible
  • Domain expertise encapsulation

Prompt Applications

  • Instructional blueprints
  • Domain-specific reasoning
  • Multi-turn conversations
  • Task-specific agents
  • Standardized output formats

Basic Prompt Implementation

Simple String Prompt
@mcp.prompt()
def ask_about_topic(topic: str) -> str:
    """Ask for an explanation of a topic."""
    return f"Can you explain the concept of '{topic}'?"

Advanced PromptMessage Implementation

Structured PromptMessage
from fastmcp import FastMCP
from fastmcp.prompts.prompt import PromptMessage, TextContent

# Prompt returning a specific message type
@mcp.prompt()
def generate_code_request(language: str, task_description: str) -> PromptMessage:
    """Generates a user message requesting code generation."""
    content = f"""Write a {language} function that
performs the following task:
{task_description}
"""
    return PromptMessage(
        role="user",
        content=TextContent(type="text", text=content)
    )

Multi-turn Conversations

Multi-turn with PromptMessage
@mcp.prompt()
def roleplay(character: str) -> list[PromptMessage]:
    return [
        PromptMessage(role="user",
                     content=TextContent(text=f"Behave like {character}.")),
        PromptMessage(role="assistant", 
                     content=TextContent(text="Understood. I'm ready."))
    ]
Simplified with Message Constructor
from fastmcp.prompts.prompt import Message

@mcp.prompt()
def roleplay(character: str) -> list[PromptMessage]:
    return [
        Message(f"You are now {character}. Respond in character."),
        Message("Understood. I'm ready.", role="assistant")
    ]

Type Annotations & Validation

Type-Annotated Prompt with Validation
from pydantic import Field
from typing import Literal, Optional

@mcp.prompt()
def content(
    topic: str = Field(description="The subject to cover"),
    formatting: Literal["blog", "email", "social"] = "blog",
    tone: str = "professional",
    word_count: Optional[int] = None
) -> str:
    """Create a request for generating content in a specific format."""
    prompt = f"Please write a {formatting} post about {topic} in a {tone} tone."
    if word_count:
        prompt += f" It should be approximately {word_count} words long."
    return prompt

Type Annotation Benefits

  • Automatic input validation from clients
  • Schema generation for MCP protocol
  • Better IDE support and documentation
  • Runtime parameter checking

When to Use Each Component

Use Tools When
  • Data is dynamic and requires real-time fetching
  • Computation needed based on user input
  • External APIs must be called with parameters
  • Actions need to be performed
  • Filtering/processing required before results
Use Resources When
  • Data is static or changes infrequently
  • Documentation needs to be referenced
  • Context needed for AI reasoning
  • Background knowledge required
  • Pre-existing datasets should be available
Use Prompts When
  • Standardized workflows need structure
  • Complex interactions need guidance
  • Domain expertise can be captured
  • User onboarding requires guidance
  • Consistent patterns should be embedded

Practical Demonstration: Job Search Assistant

Complete MCP Workflow Architecture

Tools Layer
search_jobs(), save_job()
Resources Layer
resume://default, jobs://saved
Prompts Layer
analyze_job_market(), recommendations()

Tools Implementation

  • search_jobs(): Fetches job listings using JSearch API based on role and location parameters
  • save_job(): Allows users to save specific jobs in structured format with essential information
  • Both tools store raw results and return key details for model response generation
  • Real-time data processing with dynamic parameter handling

Resources Layer

  • resume://default: User's resume content
  • jobs://saved: Saved job listings
  • Static reference material for AI reasoning
  • Structured data access without side effects

Prompts Layer

  • analyze_job_market(): Analyzes jobs for trends
  • personalized_job_recommender(): Uses resume for tailored recommendations
  • create_match_report(): Resume-to-job matching analysis

Workflow Process

  1. User inputs query through MCP host (Claude Desktop)
  2. MCP server decides which tools/resources/prompts to invoke based on query context
  3. Relevant components execute in coordinated manner
  4. Results returned to MCP host where LLM generates final response

This demonstrates the power of coordinated MCP usage - tools fetch dynamic data, resources provide context, and prompts guide the analysis workflow.

Best Practices & Configuration

Comprehensive FastMCP Server Setup
# Initialize FastMCP with all duplicate handling options
mcp = FastMCP(
    name="job-search-assistant",
    on_duplicate_tools="error",      # Prevent tool conflicts
    on_duplicate_resources="warn",   # Log resource replacements  
    on_duplicate_prompts="replace"   # Allow prompt updates
)

# All three duplicate handling settings available:
# - on_duplicate_tools
# - on_duplicate_resources  
# - on_duplicate_prompts

Development Guidelines

  • Use clear, descriptive function names and docstrings
  • Implement comprehensive error handling
  • Add detailed type annotations for validation
  • Configure appropriate duplicate handling strategies
  • Test all components thoroughly in isolation

Security Considerations

  • Validate all input parameters rigorously
  • Implement explicit user permission checks
  • Sanitize file paths and URLs properly
  • Limit resource access scope appropriately
  • Monitor and log tool execution patterns

Return Type Handling in FastMCP

  • str: Automatically converted to single PromptMessage object
  • PromptMessage: Used directly as provided
  • list[PromptMessage | str]: Used as sequence of messages (conversation)
  • Any: Attempted conversion to string, then used as PromptMessage

Real-World Use Case Examples

Software Development Assistant

  • Tools: Code generation, execution, and testing
  • Resources: Language documentation and company standards
  • Prompts: Bug analysis and code review workflows

Customer Service Platform

  • Tools: Real-time order lookup and return processing

Application-Controlled vs Model-Controlled Access Patterns

Tools (Model-Controlled):
  • LLM decides when to execute
  • User approval required
  • Real-time invocation
Resources (Application-Controlled):
  • Client fetches explicitly
  • User may select resources
  • Provided before LLM processing

Key Takeaways

MCP's Unified Approach to AI Workflows

Tools, Resources, and Prompts work together to create powerful, modular AI workflows through coordinated interaction patterns and clear separation of concerns.

Modularity

Components can be developed, maintained, and updated independently with clear interfaces and minimal interdependence.

Scalability

Easy to add new capabilities without disrupting existing functionality through URI templates and dynamic resource discovery.

Security

Clear separation of concerns with appropriate access controls, user permissions, and controlled execution patterns.

Integration Benefits

  • Consistent interaction patterns across components
  • Maintainable and testable AI workflows
  • Reusable components across different applications
  • Clear boundaries between data types and operations
  • Standardized discovery and execution mechanisms
  • Flexible duplicate handling strategies

Remember the Core Principle

Match the primitive to the specific use case based on data dynamism, required actions, and control patterns
